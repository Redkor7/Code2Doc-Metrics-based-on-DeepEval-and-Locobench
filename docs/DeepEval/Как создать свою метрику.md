### Базовый класс:

```python
from __future__ import annotations

from abc import abstractmethod
from typing import TYPE_CHECKING, Optional, Dict, List

from deepeval.test_case import (
    LLMTestCase,
    ConversationalTestCase,
    LLMTestCaseParams,
    ArenaTestCase,
)

if TYPE_CHECKING:
    from deepeval.models import DeepEvalBaseLLM


class BaseMetric:
    _required_params = List[LLMTestCaseParams]
    threshold: float
    score: Optional[float] = None
    score_breakdown: Dict = None
    reason: Optional[str] = None
    success: Optional[bool] = None
    evaluation_model: Optional[str] = None
    strict_mode: bool = False
    async_mode: bool = True
    verbose_mode: bool = True
    include_reason: bool = False
    error: Optional[str] = None
    evaluation_cost: Optional[float] = None
    verbose_logs: Optional[str] = None
    skipped = False
    requires_trace: bool = False
    model: Optional[DeepEvalBaseLLM] = None
    using_native_model: Optional[bool] = None

    def __init_subclass__(cls, **kwargs):
        super().__init_subclass__(**kwargs)
        from deepeval.tracing.internal import observe_methods
        observe_methods(cls)

    @abstractmethod
    def measure(self, test_case: LLMTestCase, *args, **kwargs) -> float:
        raise NotImplementedError

    @abstractmethod
    async def a_measure(self, test_case: LLMTestCase, *args, **kwargs) -> float:
        raise NotImplementedError(
            f"Async execution for {self.__class__.__name__} not supported yet. Please set 'async_mode' to 'False'."
        )

    @abstractmethod
    def is_successful(self) -> bool:
        raise NotImplementedError

    @property
    def __name__(self):
        return "Base Metric"

    def _accrue_cost(self, cost: float) -> None:
        if self.evaluation_cost is not None and cost is not None:
            self.evaluation_cost += cost
        else:
            self.evaluation_cost = None
```

### Параметры базового класса:
#### 1. Настройки метрики (задаются до запуска)
- **`_required_params = List[LLMTestCaseParams]`**
    - **Что это:** Список данных, которые _обязательно_ должны быть в тест-кейсе для работы этой метрики.
    - _Пример:_ Если метрика оценивает релевантность ответа контексту, ей обязательно нужны `ACTUAL_OUTPUT` (ответ) и `RETRIEVAL_CONTEXT` (контекст). Если пользователь забудет их передать, фреймворк выдаст ошибку до запуска.
- **`threshold: float`**
    - **Что это:** Порог успешности (обычно от 0.0 до 1.0).
    - _Зачем нужен:_ С этим числом будет сравниваться итоговый балл (`score`). Если балл больше или равен порогу — тест пройден (Passed).
- **`strict_mode: bool = False`**
    - **Что это:** Строгий режим оценки.
    - _Зачем нужен:_ Если включен (`True`), метрика может применять более жесткие правила. Например, вместо выдачи дробного балла (0.8), она может округлять до бинарного (0 или 1), наказывая за малейшие ошибки.
- **`async_mode: bool = True`**
    - **Что это:** Флаг асинхронности.
    - _Зачем нужен:_ Если `True`, фреймворк будет вызывать метод `a_measure` (асинхронно). Это позволяет запускать десятки тестов параллельно. Если ваша метрика не поддерживает асинхронность, нужно поставить `False`.
- **`include_reason: bool = False`**
    - **Что это:** Требование сгенерировать текстовое объяснение оценки.
    - _Зачем нужен:_ Если `True`, метрика (обычно с помощью LLM-судьи) потратит дополнительные ресурсы/токены, чтобы написать подробный текст, почему она поставила именно такой балл.

#### 2. Результаты выполнения (заполняются внутри метода `measure`)

- **`score: Optional[float] = None`**
    - **Что это:** Итоговая оценка (балл), которую вычислила метрика.
    - _Зачем нужен:_ Главный результат работы. Изначально пусто (`None`), заполняется в процессе оценки.
- **`score_breakdown: Dict = None`**
    - **Что это:** Детализация оценки (словарь).
    - _Зачем нужен:_ Если метрика сложная и состоит из нескольких под-метрик (например, оценивает грамматику, стиль и фактологию отдельно), здесь будут храниться баллы по каждому пункту.
- **`reason: Optional[str] = None`**
    - **Что это:** Текстовое обоснование оценки.
    - _Зачем нужен:_ Сюда записывается текст, объясняющий, почему модель получила свой `score` (например: _"Модель ответила правильно, но упустила деталь X"_).
- **`success: Optional[bool] = None`**
    - **Что это:** Статус успешности теста.
    - _Зачем нужен:_ Принимает значение `True` или `False` после вызова метода `is_successful()`. Определяет, будет ли тест зеленым или красным в итоговом отчете.

#### 3. LLM-судья (параметры для метрик, использующих другие нейросети для оценки)

- **`evaluation_model: Optional[str] = None`**
    - **Что это:** Строковое название модели-судьи.
    - _Зачем нужен:_ Показывает в отчетах, какая модель оценивала тест (например, `"gpt-4o"` или `"claude-3-opus"`).
- **`model: Optional[DeepEvalBaseLLM] = None`**
    - **Что это:** Сам объект (инстанс) модели-судьи.
    - _Зачем нужен:_ Через этот объект метрика отправляет промпты и получает ответы.
- **`using_native_model: Optional[bool] = None`**
    - **Что это:** Флаг использования встроенной модели.
    - _Зачем нужен:_ Показывает `True`, если метрика использует стандартную модель DeepEval (по умолчанию обычно GPT-4), и `False`, если вы передали туда свою кастомную модель (например, локальную Llama 3).
- **`evaluation_cost: Optional[float] = None`**
    - **Что это:** Стоимость выполнения метрики.
    - _Зачем нужен:_ Сюда суммируются затраты в долларах (или токенах) на API-вызовы к LLM-судье при проверке конкретно этого тест-кейса.

#### 4. Логирование, отладка и ошибки

- **`verbose_mode: bool = True`**
    - **Что это:** Режим подробного вывода в консоль.
    - _Зачем нужен:_ Если `True`, метрика будет печатать в терминал промежуточные шаги своей работы.
- **`verbose_logs: Optional[str] = None`**
    - **Что это:** Сохраненные подробные логи.
    - _Зачем нужен:_ Сюда записываются "сырые" данные (например, точный промпт, отправленный судье, и его сырой ответ). Это сохраняется в веб-отчет для дебага.
- **`error: Optional[str] = None`**
    - **Что это:** Текст ошибки выполнения.
    - _Зачем нужен:_ Если во время работы метрики произошел сбой (например, отвалился интернет при запросе к OpenAI), сюда запишется текст ошибки (Exception). Тест пометится как сломанный, но программа не упадет целиком.
- **`skipped = False`**
    - **Что это:** Флаг пропуска.
    - _Зачем нужен:_ Становится `True`, если метрику по какой-то причине не стали вычислять (например, не хватило обязательных параметров из `_required_params`).
- **`requires_trace: bool = False`**
    - **Что это:** Требование трассировки (tracing).
    - _Зачем нужен:_ Технический флаг для интеграции с системами мониторинга (например, LangSmith или Phoenix). Если `True`, фреймворк будет собирать глубокую аналитику вызовов функций внутри этой метрики.

### Шаги для создания своей метрики

#### 1. Создать класс, который наследуется от `BaseMetric`

#### 2. Написать метод `__init__` (Конструктор)

В конструкторе нужно задать базовые параметры метрики:

- `self.threshold` — порог успешности (например, 0.5 или 1.0).
- Определить `self._required_params` — список того, что обязательно должно быть в тест-кейсе (например, `[LLMTestCaseParams.INPUT, LLMTestCaseParams.ACTUAL_OUTPUT]`).

#### 3. Реализовать метод `measure` (Синхронная оценка)

Это главный метод. В нем происходит вся логика оценки.

- Он принимает `test_case` (объект с входными данными, ответом модели и ожидаемым ответом).
- Внутри надо написать логику: обращаетесь к LLM-судье, используете регулярные выражения, считаете слова и т.д.
- Важно: Внутри этого метода надо обновить атрибуты класса: `self.score` (оценка), `self.reason` (почему такая оценка) и `self.success` (успешна ли метрика, обычно вызывая `self.is_successful()`).
- Метод должен вернуть число (`float`) — итоговую оценку.

#### 4. Реализовать метод `a_measure` (Асинхронная оценка)

То же самое, что и `measure`, но для асинхронного выполнения (`async def`). Если метрика делает сетевые запросы (например, к OpenAI), асинхронность сильно ускорит тестирование.

Если вы не хотите писать асинхронный код, в `__init__` нужно явно указать `self.async_mode = False`.

#### 5. Реализовать метод `is_successful`

Этот метод просто отвечает на вопрос: "Пройден ли тест?". Обычно это простое сравнение полученного `self.score` с заданным `self.threshold`.

#### 6. Переопределить свойство `__name__`

Чтобы в логах и отчетах метрика называлась красиво, а не "Base Metric".
